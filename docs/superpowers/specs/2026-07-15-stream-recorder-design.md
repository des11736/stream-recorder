# 流收录程序设计说明

日期：2026-07-15  
状态：已确认，待实现

## 1. 目标与范围

交付一个 Windows 单文件 EXE 桌面程序。用户填写 RTMP、RTSP 或 HTTP/HTTPS 流地址后，可创建并启动多个独立收录任务。每个任务把 H.264 或 H.265 视频流封装为 HLS，输出一个 index.m3u8 播放清单及对应分片文件。

程序必须同时做到：

1. 不要求用户单独安装 Python、FFmpeg 或 FFprobe。
2. 启动前检测输入流，明确显示 H.264 或 H.265，不支持的编码不启动并给出中文原因。
3. 运行时在界面实时显示详细中文日志，并将日志保存到磁盘。
4. 记录和告警 FFmpeg 可观察到的丢包/丢帧、断流、重连失败、无媒体数据、黑屏、冻结画面、解码及写入异常。

“丢帧”是客户端基于 FFmpeg 接收、解码和时间戳警告得出的观测值；程序不把源端设备内部未对外发送的帧误报为已检测到的丢帧。

## 2. 技术方案

采用 Python 3.10+、PySide6 和 PyInstaller 构建原生 Windows 桌面程序。使用 FFmpeg 8.1.2 稳定版及同版本 FFprobe 作为随程序分发的二进制工具。主应用和 FFmpeg 进程通过标准输出、标准错误和 progress 管道通信，不使用不稳定的第三方流媒体封装库。

选择原因：

- PySide6 支持清晰的原生任务表、日志表和后台线程模型。
- FFmpeg 原生支持 RTMP、RTSP、HTTP/HTTPS、H.264、H.265 与 HLS。
- 只封装视频流，无须重编码，可减少 CPU 占用并保留原始视频质量。

交付给用户的文件只有“流收录程序.exe”。PyInstaller 的单文件模式会在程序启动时将 FFmpeg、FFprobe 和许可证临时释放到系统临时目录，退出后自动清理；用户无需管理这些文件。开发目录保留 Python 源码、测试和构建脚本。FFmpeg 的许可证、构建来源与源代码获取方式在“关于”窗口和发布说明中提供。

## 3. 界面与使用流程

主窗口包含以下区域：

1. 顶部工具栏：新建任务、启动所选、停止所选、设置输出根目录、打开日志目录。
2. 任务编辑区：任务名称、流地址、输出目录、RTSP 传输方式（自动/TCP/UDP）、分片时长、画面分析开关。
3. 任务列表：名称、协议、检测编码、状态、已收录时长、当前输出目录、最新事件、启动/停止/打开目录按钮。
4. 实时日志区：显示当前选中任务的中文日志，支持自动跟随、暂停滚动、信息/警告/错误筛选、复制、清空界面缓存与打开完整文件。

创建任务时地址框会自动识别协议；允许 rtmp://、rtmps://、rtsp://、http:// 和 https://。用户最初所写的 “rmtp” 按标准协议解释为 RTMP。地址可含鉴权信息，但界面、日志和任务列表会遮蔽密码。

默认输出结构如下：

    输出根目录/
      任务名称_20260715_140621/
        index.m3u8
        init.mp4
        segment_000001.m4s
        segment_000002.m4s
        logs/
          应用日志-2026-07-15.log
          ffmpeg-原始输出.log

单击“启动”后，流程为：校验地址 -> FFprobe 探测 -> 校验视频编码 -> 创建输出目录 -> 启动收录进程和可选画面分析进程 -> 持续显示状态与日志。单击“停止”会优雅终止进程并保留已生成的 m3u8 和分片。

## 4. 收录与 HLS 生成

### 4.1 输入探测

FFprobe 读取输入流的首个视频流和可选音频流，提取协议、codec_name、分辨率、帧率和音频编码。视频只允许 h264 和 hevc；hevc 在界面显示为 H.265。探测超时、无视频流和不支持编码均会阻止任务启动。

### 4.2 FFmpeg 主收录进程

视频使用 stream copy：-c:v copy。若输入音频为 AAC 则复制；其余可解码音频转为 AAC，输入没有音频时正常收录纯视频。输出采用 fMP4 HLS，以便同时稳定承载 H.264 和 H.265：

- 分片时长默认 6 秒，用户可设为 2 至 60 秒。
- 清单文件固定为 index.m3u8。
- 使用独立分片、临时文件和节目时间戳，避免播放器读到未写完的分片。
- 播放清单保存所有本次收录分片；重连时追加清单并插入不连续标记。
- 分片编号采用时间源，避免重连覆盖原有分片。

主命令包含只读输入超时、损坏包丢弃、实时进度输出与可解析的 warning/error 日志。RTSP 的默认传输方式为 TCP，用户可以在任务设置改为 UDP 或自动。

### 4.3 断流与重连

连续 15 秒没有进度或没有有效媒体数据时记录“流信号中断”。收录进程异常退出时记录退出码及解析出的原因。默认自动重连 5 次，退避间隔为 2、4、8、16、30 秒；成功后记录“重连成功”。超过次数后任务进入“失败”，用户可以手动重新启动。

## 5. 画面健康分析与中文日志

视频复制本身不解码，无法可靠识别黑屏或静止画面。因此在“画面分析”开启时，程序会为该任务再启动一个低帧率分析 FFmpeg 进程；它只解码视频并输出到空接收器，不写入文件，也不影响主收录文件。该进程会额外建立一个源连接。

分析规则：

| 类别 | 条件 | 中文事件 |
| --- | --- | --- |
| 接收端丢帧/丢包 | FFmpeg 输出包过期、队列溢出、丢弃帧、时间戳异常或解码警告 | 检测到接收端丢帧/丢包，并附本周期数量或原始原因 |
| 断流 | 15 秒无进度、I/O 超时、连接关闭或进程异常退出 | 流信号中断；开始第 N 次重连 |
| 无有效画面 | 分析进程在启动宽限期后未解出视频帧 | 原始视频信号无有效画面 |
| 黑屏 | blackdetect 判定连续黑帧达 10 秒 | 原始视频画面持续黑屏 N 秒 |
| 冻结 | freezedetect 判定连续静止画面达 15 秒 | 原始视频画面冻结 N 秒 |
| 编码异常 | 探测失败、非 H.264/H.265、解码器错误 | 视频编码不受支持或视频流解析异常 |
| HLS 写入异常 | 清单/分片写入失败、空间不足、权限不足 | HLS 文件写入失败，并给出路径或系统错误 |

日志分为“应用中文日志”和“FFmpeg 脱敏原始输出”两类，均为 UTF-8。每条应用日志格式为：

    [时间] [级别] [任务名] [类别] 描述；建议处理方式

例如：

    [2026-07-15 14:07:09.462] [错误] [摄像头-01] [断流] 15 秒未接收到有效媒体数据；正在第 1/5 次自动重连

实时日志通过线程安全队列发送到界面，单条日志即时显示且写入文件。界面只缓存最近 10,000 条，完整日志不截断。密码、签名令牌等 URL 敏感内容在两类可见日志中均替换为掩码。

## 6. 软件模块

1. models：任务、状态、探测结果、日志事件的数据模型。
2. config_store：保存输出根目录、任务草稿和用户偏好；启动时恢复未运行任务。
3. ffmpeg_tools：定位内置二进制、构建 FFprobe/FFmpeg 命令、解析探测 JSON。
4. recorder：管理单个主收录进程、进度、重连和 HLS 输出。
5. health_monitor：管理可选画面分析进程，识别黑屏、冻结、无画面和协议异常。
6. event_logger：解析原始输出为中文事件，脱敏并同步写入实时队列、应用日志和原始日志。
7. task_controller：协调多个独立任务，保证停止、清理和重连不会阻塞界面线程。
8. ui：任务编辑、任务表、实时日志与设置对话框。

每个收录任务拥有独立进程、队列、输出目录和日志文件；一个任务错误不会停止其他任务。

## 7. 错误处理与安全

- 输入为空、协议不支持、输出目录不可写、磁盘空间不足、FFmpeg 缺失、地址无法连接、编码不支持都会给出中文错误。
- 密码只在内存和经操作系统保护的本地配置中保存；显示和日志一律脱敏。
- 退出应用时先请求停止全部任务，等待短暂期限后再终止子进程，避免损坏正在写入的清单。
- 不提供或绕过任何 DRM、鉴权或访问控制；用户仅可收录其有权访问的流。

## 8. 测试与验收

单元测试覆盖地址识别、URL 脱敏、FFprobe 解析、H.264/H.265 校验、HLS 命令构建、FFmpeg 事件中文化、黑屏/冻结事件解析、重连状态机和日志文件写入。

集成测试使用本地 FFmpeg 生成的 H.264、H.265 测试流以及本地 HTTP/RTSP/RTMP 测试端点，验证：

1. 三种协议均能创建任务并输出 index.m3u8。
2. H.264 和 H.265 输入均通过探测并生成可解析的 HLS 清单和 fMP4 分片。
3. 中断源流时实时日志显示断流和重连；恢复后显示重连成功。
4. 人工黑屏/静止测试流触发对应中文日志。
5. 模拟 FFmpeg 警告时解析为接收端丢帧/丢包中文事件。
6. 两个任务同时运行时，停止其中一个不影响另一个。
7. PyInstaller 产物在没有开发环境的目录中能找到内置 FFmpeg 并成功启动。

验收以实际 EXE、自动化测试结果和一次本地端到端收录结果为准。
